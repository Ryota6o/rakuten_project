## 0. 毎週の共通ルール

- **常にデプロイ可能に保つ**：W1で初デプロイ後、各層を差し替えながら緑を維持する（統合を終盤にまとめない）。
- **小さく刻む**：AIには小さくレビュー可能な単位で生成させる。巨大な塊を一度に作らせない。
- **検証の柱**：配管側はAIにテストを書かせ、中核（pipeline）は `eval/metrics.py` の指標をテストハーネスとする。
- **意思決定ログ**：主要な設計判断（閾値・手法選択など）を理由つきで記録する（README・面接準備を兼ねる）。
- **AI生成の固有注意**：APIフィールドの幻覚、旧エンドポイント、鍵の直書きを毎回チェックする。
- **役割分担**：広さ（配管・UI・デプロイ）はAI、深さ（名寄せ・評価・設計判断）は自分が握る。

## フェーズ1：中核（エンティティ解決）｜W1–W6（6/27〜8/7）

### W1（6/27–7/3）最薄スライス＋初デプロイ ◆

- **目的**：精度ではなく、統合・デプロイの不確実性を初週に消す。
- **主タスク**
  - 楽天アプリ登録（**バックエンドサービス型**、許可IPに自分のグローバルIP）、`accessKey` 取得、`.env` 管理
  - フォルダツリー一式をスタブで作成
  - `ingest`：ヘッドホン数百件を取得（1.5秒スロットリング）→ DB保存
  - `matching`：**型番の完全一致のみ**／`pricing`：価格＋送料フラグのみ
  - `backend`：`/search` 1本／`frontend`：素のHTMLテーブル
  - デプロイ：backend（Render/Railway）＋front（Vercel）を緑に
  - スパイク：content scriptが楽天DOMから商品特定でき、CORSでbackendを呼べるか（30分）
- **AIに任せる**：足場、デプロイ設定、APIクライアント雛形
- **自分が握る**：アプリ種別とIP/CORSの確認、型番抽出の最初の正規表現
- **完了の定義**：「検索→型番一致→比較表」が公開URLで動作する
- **マイルストーン**：最薄スライスが公開URLで動作
- **目安時間**：20–25h

### W2（7/4–7/10）正規化・型番抽出＋評価セット着手

- **目的**：マッチングの土台（きれいな属性）と、精度を測る物差しの準備。
- **主タスク**
  - SudachiPy導入、ノイズ除去（【送料無料】等）
  - 型番・ブランド・容量の抽出、`attributeFlag=1` の構造化属性を活用
  - **「同一商品」の定義（アノテーションガイドライン）を確定**（色違い・容量違い・セット品の扱い）
  - 評価セットのラベル付けを開始
- **AIに任せる**：SudachiPyの定型処理、正規表現の叩き台
- **自分が握る**：「同一商品」の定義、抽出ロジック
- **完了の定義**：正規化後の属性が確認でき、評価セットの定義が固まり一部ラベル済み
- **目安時間**：15–20h

### W3（7/11–7/17）評価セット完成＋ブロッキング

- **目的**：候補ペアを現実的な計算量に絞り、測れる状態にする。
- **主タスク**
  - 200–500ペアの正解ラベルを完成
  - ブロッキング（ブロックキー＋FAISS近傍）を実装
  - 候補削減率と再現率（取りこぼし）を記録
- **AIに任せる**：FAISS/hnswlibの実装雛形
- **自分が握る**：ブロッキング戦略、再現率とのトレードオフ評価
- **完了の定義**：評価セット完成、ブロッキングが候補ペアを出力し、削減率／再現率が出る
- **目安時間**：15–20h

### W4（7/18–7/24）ベースライン名寄せ＋初回指標 ◆

- **目的**：数値で語れる最初の状態をつくる。
- **主タスク**
  - 語彙特徴（編集距離・Jaccard）＋型番一致でベースライン判定
  - precision/recall/F1 を算出、`eval/metrics.py` を整備
- **AIに任せる**：特徴量計算の実装、評価ループの雛形
- **自分が握る**：特徴量設計、指標の解釈
- **完了の定義**：評価セットに対して P/R/F1 が出る
- **マイルストーン**：測定可能なベースライン
- **目安時間**：15–18h

### W5（7/25–7/31）埋め込み＋分類器＋アブレーション

- **目的**：精度を上げ、寄与を分解して語れるようにする。
- **主タスク**
  - 日本語埋め込み（multilingual-e5 / Ruri）でコサイン類似
  - LightGBM分類器でマッチング
  - アブレーション（語彙のみ → ＋埋め込み → ＋型番一致）
- **AIに任せる**：sentence-transformers / LightGBM の定型コード
- **自分が握る**：分類器の特徴量と閾値、アブレーションの設計と解釈
- **完了の定義**：ベースライン比で指標が改善し、アブレーション表が出る
- **目安時間**：15–20h

### W6（8/1–8/7）クラスタリング＋実質価格

- **目的**：パイプラインを一気通貫させる（同一商品グループ＋実質価格）。
- **主タスク**
  - 連結成分＋閾値で商品クラスタ化、推移律破綻の扱いを決める
  - 実質価格モデル（価格＋送料フラグ−ポイント簡易推定）
- **AIに任せる**：networkx/scipy の実装、価格計算の実装
- **自分が握る**：クラスタリング手法と閾値、実質価格の簡易化方針の明示
- **完了の定義**：「商品クラスタ＋店舗別の実質価格」がDBに出力される
- **マイルストーン**：パイプライン一気通貫
- **目安時間**：15–18h

## フェーズ1：Webアプリ｜W7–W8（8/8〜8/21）

### W7（8/8–8/14）API＋画面の骨組み

- **目的**：パイプラインの成果をWebで見せる土台をつくる。
- **主タスク**
  - FastAPIで「検索→クラスタ→実質価格」のAPI、Pydanticで応答契約
  - Reactで検索＆テーブルの骨組み（backendは `pipeline` を import）
- **AIに任せる**：FastAPI/Reactの実装、API結線
- **自分が握る**：API設計（応答契約）、どの指標・情報を見せるか
- **完了の定義**：ローカルで「検索→比較テーブル」が本データで動く
- **目安時間**：15–18h

### W8（8/15–8/21）UI仕上げ＋デプロイ ◆

- **目的**：v1完成＝面接で提示できる状態にする。
- **主タスク**
  - 比較テーブルUIの仕上げ（実質価格ソート、店舗／ポイント／レビュー表示、「◯時点」表記）
  - デプロイ（Render/Railway ＋ Vercel）、本番ドメインのAPI登録対応
- **AIに任せる**：UI実装、デプロイ設定
- **自分が握る**：完成の定義の充足確認、見せ方
- **完了の定義**：公開URLでv1が動作する
- **マイルストーン**：v1デプロイ＝面接で提示可能
- **目安時間**：12–18h

## フェーズ2：拡張機能・仕上げ｜W9–W10（8/22〜9/6）

### W9（8/22–8/28）Chrome拡張

- **目的**：楽天ページ上での比較表示（コアへの上乗せ）。
- **主タスク**
  - Manifest V3の骨組み
  - content scriptで閲覧中商品を特定 → backendのクラスタに対応づけ
  - オーバーレイUI（同一backendを呼ぶ）
- **AIに任せる**：MV3ボイラープレート、content script、CORS
- **自分が握る**：商品特定 → クラスタ照合のロジック
- **完了の定義**：楽天商品ページで「他店だと実質◯円」が重なって表示される
- **マイルストーン**：楽天ページ上での比較表示
- **目安時間**：12–15h

### W10（8/29–9/6）仕上げ・予備 ◆

- **目的**：最終成果物として提出できる状態にする。
- **主タスク**
  - DOM例外処理
  - README（指標・アブレーション・制約・**設計判断＝意思決定ログ**）
  - デモ録画、予備日
- **AIに任せる**：文章整形・リファクタ補助
- **自分が握る**：READMEでの設計判断の言語化（面接準備）
- **完了の定義**：README完備、デモ録画あり、全体が公開URLで動作する
- **マイルストーン**：最終成果物の完成
- **目安時間**：8–12h

- 隠れ工数の最大は **評価セットのラベル付け（W2–W3で合計8–15h）**。甘く見ると後ろが押す。
