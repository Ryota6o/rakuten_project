## 1. 背景と目的

### 1.1 解こうとしている課題

楽天市場はマーケットプレイス型のため、**同一商品が多数の店舗から個別に出品され、価格・送料・ポイント・クーポンが店ごとにバラつく**。利用者が「実質的に最も安い出品」を見つけるには、出品を一つずつ手で比較する必要があり、負担が大きい。

楽天は2026年1月より対話型の「Rakuten AI（AIコンシェルジュ）」を市場アプリに搭載し、発見・提案（ディスカバリー）領域には投資している。しかし、**「同一商品を店舗横断で正規化し、実質価格で比較する」という構造的課題は未解決**である。本アプリはこの残された課題、すなわち「比較・正規化・信頼性」の領域を対象とする。

### 1.2 なぜAmazon横断ではなく楽天店舗横断なのか

- AmazonのデータをProduct Advertising API（PA-API）で取得するには売上実績が必要で、個人開発では実質利用できない。モバイルアプリ内でのPA-APIデータ利用も禁止されている。直接スクレイピングはToS違反かつbot対策で非現実的。
- 一方、楽天は**楽天ウェブサービス（楽天商品検索API）をアプリID登録のみで無料公開**しており、価格・送料・ポイント・店舗・型番・属性が取得できる。
- 「楽天の課題を楽天の公式APIで解く」構成のため、楽天の面接における志望動機ストーリーとも整合する。

> Amazon横断は将来拡張の余地として位置づけ、本仕様の対象スコープからは外す。

## 2. スコープと開発方針

開発は2フェーズに分割する。**フェーズ1で独立Webアプリとして中核を完成・デプロイし、フェーズ2で同じバックエンドを呼ぶ薄い拡張機能を載せる**。この順序により、途中で時間切れになっても「完成してデプロイ済みのWebアプリ」が必ず手元に残る。

| フェーズ | 成果物 | 内容 |
| --- | --- | --- |
| フェーズ1 | 独立Webアプリ | 名寄せ（エンティティ解決）＋実質価格算出＋比較テーブル＋評価指標。技術的中核はすべてここに置く |
| フェーズ2 | Chrome拡張機能 | 楽天市場ページにオーバーレイ表示する「見せ方レイヤー」。バックエンドはフェーズ1を再利用 |

### 2.1 完成イメージ

- **フェーズ1（Webアプリ）**：利用者がアプリのサイトで商品を検索すると、同一商品の店舗別出品が実質価格の安い順で並ぶ比較テーブルを表示する。
- **フェーズ2（拡張機能）**：利用者が楽天市場の商品ページを開いて拡張機能をONにすると、ページ上に「他店だと実質◯円」というパネルが重なって表示され、楽天の見た目が拡張される。重ねて表示する比較データは、楽天ページのスクレイピングではなく、バックエンドが公式APIで集めて返す。

### 2.2 開発アプローチ：最薄スライス先行 × AI駆動開発

本開発は「層を下から順に積む」のではなく、**最初に端から端まで貫く最薄の実装（ウォーキングスケルトン）を通し、初デプロイまで済ませてから、各層を継続的に差し替えて作り込む**（継続統合）。これにより統合・デプロイの不確実性を初週に潰し、ビッグバン統合や「名寄せの沼にはまって動くアプリが無い」状態を回避する。詳細は §13。

実装は**AI駆動開発**で進める。広さ（配管・UI・デプロイ・拡張機能の定型部分）はAIに任せ、深さ（名寄せロジック・評価設計・アーキテクチャと設計判断）は自分が握る。役割分担と生成コードの検証ルールは §14。着手前に潰すべき確認事項は §15。

## 4. システム構成

### 4.1 処理パイプライン（フェーズ1）

中核はエンティティ解決であり、ブロッキング・マッチング・クラスタリングの3工程が技術的な見せ場となる。

```
データ取得（楽天商品検索API）
   ↓
正規化・属性抽出（型番・ブランド・容量を抽出）
   ↓
ブロッキング（候補ペアを絞り込む）            ← エンティティ解決の中核
   ↓
マッチング（語彙＋意味＋型番で一致判定）       ← エンティティ解決の中核
   ↓
クラスタリング（同一商品グループへ統合）       ← エンティティ解決の中核
   ↓
実質価格の算出（価格＋送料−ポイント）
   ↓
Web UI（実質価格順の比較テーブル）
```

### 4.2 拡張機能のデータフロー（フェーズ2）

```
楽天の商品ページを開く（ユーザーが商品を閲覧）
   ↓ 表示中の商品を読み取る
拡張機能（content script）が商品を特定
   ↓ 商品キーを問い合わせ
バックエンドで名寄せ（楽天APIで横断検索）
   ↓ 比較結果（実質価格の安い順）を返す
拡張機能がページに重ねて表示（楽天の見た目が変わる）
```

> 重要：オーバーレイに表示する比較データは、楽天ページのDOMから作るのではなく、バックエンドが公式APIで取得・名寄せして生成する。拡張機能が楽天ページから読み取るのは「いま見ている商品が何か」だけである。

## 5. データ仕様（楽天商品検索API）

### 5.1 エンドポイントと認証

- 最新版エンドポイント：`https://openapi.rakuten.co.jp/ichibams/api/IchibaItem/Search/20260401`
- 認証：`applicationId` に加え、`accessKey` が必須（最新版で追加）。
- レスポンス形式：`formatVersion=2` を指定し、フラットなJSONで扱う。

### 5.2 利用する主なレスポンス項目

| 用途 | 項目 |
| --- | --- |
| 同定・マッチング | `itemName`（商品名）、`catchcopy`（キャッチコピー）、`itemCaption`（商品説明）、`itemCode`、`genreId`、`shopName` / `shopCode`、`attributes` / `attributeId`（`attributeFlag=1` 指定時の構造化属性） |
| 実質価格 | `itemPrice`、`taxFlag`、`postageFlag`、`pointRate`（＋ `pointRateStartTime` / `pointRateEndTime`）、`startTime` / `endTime`（セール期間）、`availability`、`creditCardFlag` |
| 品質・信頼性 | `reviewCount`、`reviewAverage` |
| 画像・その他 | `smallImageUrls`（64px）、`mediumImageUrls`（128px、各最大3枚）、`itemUrl`、`shopUrl`、`giftFlag` |

### 5.3 データ上の制約（仕様で割り切る点）

これらはAPIの仕様上の制約であり、設計判断として明示的に割り切る。READMEにも記載する。

| 制約 | 内容 | 対応方針 |
| --- | --- | --- |
| 送料の実額が取れない | `postageFlag` は「送料込み／無料か否か」のフラグのみで、送料の円額は返らない | 実質価格の送料成分は「無料／有料」フラグで扱い、円額は推定または割り切る旨を明示する |
| ポイントが限定的 | 通常購入で1%。APIが対応するのは商品単位のポイント倍率のみで、ショップ単位の倍率やSPUは含まれない | 期待ポイントは「基本1%＋商品単位倍率」の簡易モデルとし、近似である旨を明示する |
| レビュー本文なし | 取得できるのは件数（`reviewCount`）と平均（`reviewAverage`）のみ。本文は返らない | 本アプリでは件数・平均のみ使用。サクラ検出など本文を要する機能は対象外とする |
| 画像が低解像度 | 最大128pxのサムネイルのみ | 画像ベースの照合は補助シグナルにとどめ、主軸にはしない |
| 取得件数の上限 | 1ページ最大30件・最大100ページのため、1クエリで取得できるのは最大3,000件 | カテゴリ・キーワードを分割して取得対象を確保する |
| レート制限 | 短時間に同一URLへ大量アクセスすると一時的に応答停止。上限超過時は429 | スリープを挟んでスロットリングする |

## 6. 処理パイプライン詳細

### 6.1 正規化・属性抽出

楽天の商品名は「【送料無料】【ポイント10倍】メーカー名 型番 …」のようにノイズが多い。ここで**型番・ブランド・容量などの構造化属性を抽出**する。型番の一致は「同一商品」の最強シグナルであり、この工程の精度が全体精度を大きく左右する。`attributeFlag=1` で得られる構造化属性も補助に使う。

### 6.2 ブロッキング

全商品の総当たり（O(n²)）を避けるための候補絞り込み。以下を組み合わせる。

- ルールベースのブロックキー（カテゴリ、ブランド、型番プレフィックス）
- 軽量埋め込み＋近傍探索（FAISS / hnswlib）による「似ていそうなペア」の抽出

評価観点として、候補削減率と再現率（取りこぼし）のトレードオフを記録する。

### 6.3 マッチング

候補ペアごとに複数の特徴量を作り、分類器で「一致確率」を出力する。

- 語彙類似：正規化タイトルの編集距離・トークンJaccard
- 意味類似：日本語埋め込みのコサイン類似（multilingual-e5 など）
- 構造一致：型番・属性の一致フラグ
- 分類器：LightGBM または scikit-learn（堅実さ優先）。研究の見せ場として小規模な Flax MLP に差し替える選択肢も残す

### 6.4 クラスタリング

ペア判定の結果を商品グループへ統合する。閾値つきの連結成分（networkx / scipy）を基本とし、推移律の破綻（A=B、B=C だが A≠C となる矛盾）の扱いを設計上の論点として明記する。

### 6.5 実質価格の算出

```
実質負担額 = itemPrice + 送料成分 − 期待ポイント還元
  送料成分     : postageFlag（無料/有料）に基づく簡易値
  期待ポイント : itemPrice × (基本1% + 商品単位ポイント倍率)
```

送料実額とSPUがAPIで取得できないため、簡易モデルである旨を画面およびREADMEに明示する。

## 8. 評価方法

本アプリの差別化点は、名寄せ精度を**定量的に測れること**にある。多くの応募者は既製APIを呼ぶだけで指標を持たないため、ここが決め手となる。

- **評価セット**：手作業で200〜500ペアの正解ラベル（同一／非同一）を作成する。
- **指標**：マッチングの適合率（precision）・再現率（recall）・F1を報告する。
- **アブレーション**：「語彙のみ → ＋意味埋め込み → ＋型番一致」と特徴量を積み上げ、各段階の指標を比較する。
- **ブロッキング評価**：候補削減率と再現率のトレードオフを記録する。

## 10. 対象カテゴリ

初期カテゴリは、型番が明確で複数店舗に散らばる商品が名寄せの題材として扱いやすい。

- **推奨：ヘッドホン／イヤホン**。型番がはっきりしており、型番完全一致という強いシグナルでベースラインが安定する。一方でケース・ケーブルなどのノイズをNGワード（`NGKeyword`）で弾く処理も見せられ、難度が適切。音響への関心とも接続する。
- 代替候補：プリンタ互換インク、特定のPC周辺機器（SSD・モニター）。型番ノイズが少なく、定番の名寄せ素材。

## 11. 制約・リスク

| リスク | 内容 | 緩和策 |
| --- | --- | --- |
| API取得件数の上限 | 1クエリ最大3,000件 | カテゴリ・キーワードを分割して取得 |

## 13. 開発アプローチ詳細：W1 最薄スライス（ウォーキングスケルトン）＋初デプロイ

目的は**精度ではなく不確実性の除去**。W1で各層を最も貧弱な実装で一度だけ端から端まで通し、初デプロイまで到達する。

最薄スライスの構成（各層の最小実装）：

| 層 | W1の最小実装 |
| --- | --- |
| データ取得 | ヘッドホンを少量（数百件）だけ取得 |
| 正規化 | ほぼ無し（商品名そのまま） |
| マッチング | **型番の完全一致のみ**（埋め込み・分類器なし） |
| クラスタリング | 型番一致でグループ化 |
| 実質価格 | 価格＋送料フラグのみ（ポイントは後回し） |
| UI | 素のHTMLテーブル |
| デプロイ | バックエンド＋フロントを各1回デプロイし疎通（緑）にする |

規律：

- **時間を区切る**：W1内に収め、精度の深追いをしない。
- **スライスは足場であって製品ではない**：W2以降で各層を本実装に差し替える前提とし、ハックしたコードを放置しない。
- フェーズ2最大の未知（content scriptが楽天実DOMから商品を特定でき、自分のバックエンドをCORS越しに呼べるか）を、30分のスパイクで先に確認しておく。

## 14. AI駆動開発の方針（役割分担・検証ルール）

### 14.1 役割分担

| AIに大きく任せる（広さ） | 自分が握る（深さ・説明責任） |
| --- | --- |
| FastAPI/Reactの足場、比較テーブルUI | アーキテクチャ、モジュール境界、データ契約 |
| APIクライアント、スロットリング、デプロイ設定 | 名寄せロジック（ブロッキング・特徴量・クラスタリング）の設計判断 |
| Chrome拡張のボイラープレート、CORS処理 | 評価設計（評価セット作成、precision/recall/F1、アブレーション） |
| テストの叩き台、リファクタ | 「同一商品が複数店舗に価格差で現れるか」等のデータ判断 |

理由：面接で深掘りされるのは右側である。右側を自分の言葉で説明できることが、AI駆動開発のポートフォリオを成立させる条件となる。AIネイティブ開発自体は事業会社で推進されており、「どう設計・分解・レビュー・検証して使いこなしたか」を語れることはむしろ評価対象となる。

### 14.2 生成コードの検証ルール

- 本仕様書をAIへのコンテキスト（指示書）として渡す。
- 小さく刻んでレビュー可能な単位で生成する（巨大な塊を一度に生成しない）。
- 検証の柱：配管側はAIにテストを書かせ、パイプライン側は**評価セット＋指標をテストハーネスとする**。
- 主要な設計判断は**意思決定ログ**に理由つきで記録する（README・面接準備を兼ねる）。

### 14.3 このプロジェクト固有の注意（AIが間違えやすい点）

- **APIフィールドの幻覚**：楽天APIのフィールド名を創作しがち。実物の項目（`postageFlag` / `pointRate` / `attributes`）と必ず突き合わせる。
- **古いエンドポイント**：旧版を出しがち。最新版（`openapi.rakuten.co.jp` の `20260401`、`accessKey` 必須）を使っているか確認する。
- **APIキーの直書き**：キーをコードに埋め込んでリポジトリに入れない。`.gitignore` と環境変数を徹底する。
